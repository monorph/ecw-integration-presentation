01
The Problem

We Can't See Our Own Data

eClinicalWorks holds all our clinical, scheduling, and billing data — but there's no automated way to extract it for reporting.

  • No automated reporting pipeline — all reports are manual exports or eCW built-in views
  • No executive dashboard — Felipe has no real-time visibility into encounters, no-shows, or revenue
  • No cross-system analytics — data is siloed inside eCW with no easy way to combine or transform
  • Provider productivity is opaque — no standardized scorecard or trend tracking
  • Revenue cycle is a black box — AR aging, denials, and collections require manual spreadsheet work

What We Need

  • Executive dashboard — Felipe's #1 priority
  • Provider scorecards — encounters, CPTs, utilization
  • Scheduling analytics — no-show rates, fill rates
  • Revenue cycle visibility — charges, collections, AR
  • Clinical quality metrics — HEDIS/MIPS readiness

Goal: Real-time, automated dashboards that update themselves — no manual exports.

02
Options Evaluated

Three Paths — One Winner

We evaluated three approaches to pull data from eCW. Felipe approved HL7v2 on March 20.

Selected

Option 1: HL7v2

  • Most mature path for eCW
  • Real-time message feeds (ADT, SIU, DFT, ORU)
  • Proven with Mirth Connect
  • 3-4 month eCW provisioning lead time

Verdict: Best ROI. Start here.

Option 2: FHIR R4

  • Modern API standard
  • Bulk $export for clinical data
  • Rate-limited (250 calls/min)
  • eCW FHIR is still maturing

Verdict: Phase 2 supplement for quality metrics.

Option 3: Healow / eBO

  • Healow Open Access portal
  • eBO = built-in reporting/BI
  • License status unknown
  • Limited customization

Verdict: Supplement for revenue cycle data.

06
Risk Awareness

eCW-Specific Gotchas

eCW is not plug-and-play. The board should understand these constraints upfront.

Critical Constraints

  • No self-serve interfaces — everything goes through eCW Interface Services with 4-8 week lead time per interface
  • No public sandbox — testing only during eCW's provisioned testing window
  • VPN required — cloud-hosted eCW mandates IPsec tunnel (they control the endpoint)
  • ACK format must be exact — malformed ACKs cause interface suspension requiring eCW manual intervention

Revenue Cycle Gap

  • Revenue data NOT available via HL7v2 — AR aging, denials, payments require eBO or direct DB (likely unavailable)
  • DFT (charges) may be a separate license add-on — must confirm with eCW before assuming it's included

Data Quirks to Plan For

  • SIU no-show messages may arrive hours after appointment time
  • DFT charges may arrive in end-of-day batches, not real-time
  • ORU LOINC codes may be missing — need local code mapping table
  • Provider IDs in AIP segment may be eCW internal, not NPI
  • Healow portal bookings may not generate SIU in older eCW versions
  • FHIR rate limit: 250 calls/min — tight for historical backfill

Mitigation: We develop and test against synthetic data while waiting for eCW provisioning. By the time they're ready, we're ready.

Compliance

HIPAA & Security

Built-In Safeguards

  • All HL7v2 over encrypted VPN tunnel — eCW requirement, HIPAA compliant
  • PostgreSQL encrypted at rest — LUKS/FileVault on homelab
  • SSL connections enforced for all database access
  • Metabase internal-only — behind VPN, HTTPS with valid cert, no public exposure
  • pgaudit enabled for database-level audit trail
  • No PHI on public internet — ever
  • 6-year data retention — Florida medical records requirement

Existing Agreements

  • BAA with eCW — already in place
  • No additional BAAs needed — all infrastructure is on-premise (homelab)
  • No cloud PHI — data stays within AMD's physical network

This architecture was specifically designed to avoid cloud PHI complexity. Everything runs on hardware we own, in a building we control. The only external connection is the eCW VPN tunnel — which they mandate and encrypt.
